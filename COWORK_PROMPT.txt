== PROJECT SUMMARY ==

This is a software capstone project building an open-source, hardware-lightweight
pipeline for selective video compression of static surveillance camera footage for
Navy and DoD land surveillance use cases.

The core technical idea: static surveillance cameras produce massive amounts of
redundant data because most pixels in any given frame are identical to the previous
frame. By applying background subtraction to isolate only the foreground objects of
interest (people walking, vehicles), and compressing only those regions at high
quality while discarding or heavily compressing the unchanging background pixels,
the system achieves approximately 6x storage reduction versus full-frame compression.

The system stores compressed footage locally for approximately one week before it is
offloaded to longer-term storage. The compression must be reversible and the footage
must be enhanceable after offload using image/video enhancement algorithms.


== WHAT THE SPONSOR CONFIRMED IN THE KICKOFF MEETING ==

1. STATIC CAMERAS: This project focuses on static (fixed-position) surveillance
   cameras, not drone footage. This is a major scope clarification from the original
   project title which mentioned outdoor imagery broadly.

2. STORAGE FIRST, STREAMING SECOND: The primary problem is storing data efficiently
   for approximately one week while waiting for offload. Real-time transmission is
   secondary to the storage problem.

3. FOREGROUND-ONLY COMPRESSION: The sponsor explicitly described saving 6x storage
   by storing only the people and vehicles passing through the frame rather than the
   entire video. Background pixels that do not change should be discarded or heavily
   compressed.

4. LOW-SPEC HARDWARE REQUIREMENT: The solution must run on old, cheap hardware
   without modern GPUs or high RAM. The DoD and open-source context means no
   dependency on NVIDIA A100s or Jetson Orins at scale. Think: Raspberry Pi,
   commodity x86 boxes, existing embedded hardware already deployed.

5. OPEN SOURCE MANDATORY: All components, codecs, and libraries must be open-source
   and royalty-free. No proprietary solutions.

6. PEOPLE AND VEHICLES ARE THE TARGETS: The objects of intelligence value are
   people walking and people in cars. The system should prioritize preserving the
   quality of those regions specifically.

7. ENHANCEMENT IS IN SCOPE: Algorithms that take the compressed (possibly degraded)
   footage and enhance/upscale it after offload are explicitly in scope. This covers
   super-resolution and decompression enhancement for imagery that was saved at lower
   quality.

8. NAVY LAND SURVEILLANCE CONTEXT: Primary use case is Navy land surveillance
   installations. The data pipeline is: capture -> compress -> store locally ~1 week
   -> offload -> analyze/enhance.

9. SOFTWARE FOCUS: This is primarily a software project. The team is not expected to
   build hardware or integrate with classified systems. The deliverable is a working
   open-source software pipeline.


== REVISED TECHNICAL ARCHITECTURE ==

Input:
  Static camera feed (USB, IP camera, or pre-recorded video file for testing)

Stage 1 - Background Subtraction:
  OpenCV MOG2 or KNN algorithm
  Detects foreground pixels (moving objects) vs. background (static scene)
  Outputs binary foreground mask + bounding regions per frame

Stage 2 - Selective Encoding:
  Foreground ROIs (regions of interest): encoded at low CRF (high quality) with libx264
  Background: encoded at high CRF (heavy compression) or as periodic static keyframes only
  Tool: FFmpeg with libx264, open source, royalty-free, CPU-only

Stage 3 - Metadata Indexing:
  SQLite database records timestamps, camera ID, whether targets were detected,
  ROI count, file size, and segment duration
  Enables fast retrieval: "give me all segments with detected targets from camera 3 this week"

Stage 4 - Storage / Offload:
  Compressed segments accumulate locally for ~1 week
  Offloaded to longer-term storage or analysis pipeline

Stage 5 (Post-Offload) - Enhancement:
  Lightweight super-resolution on CPU (Real-ESRGAN or BSRGAN in CPU mode)
  Applied to foreground ROIs to recover detail lost during compression
  Also applicable to old/legacy footage for retroactive quality improvement

== HOW TO INITIALIZE THE GITHUB REPOSITORY ==

Run the following commands from inside the capstone-compression/ directory:

  git init
  git add .
  git commit -m "Initial commit: project scaffold with background subtraction, ROI encoder, and pipeline"
  git branch -M main
  git remote add origin https://github.com/Blood-Dawn/capstone-compression.git
  git push -u origin main

Recommended branch strategy:
  main          -- stable, always working
  dev           -- integration branch, merge features here before main
  feature/xxx   -- individual feature branches per team member

Example feature branches:
  feature/background-subtraction-tuning
  feature/roi-ffmpeg-encoding
  feature/enhancement-superresolution
  feature/benchmarking-visdrone
  feature/metadata-database


== HOW TO WORK WITH THE FILES IN THIS PROJECT ==

To run the full pipeline on a test video:
  python src/pipeline/pipeline.py --input data/samples/test_clip.mp4 --camera-id cam_test --output outputs/

To run on a live camera (index 0):
  python src/pipeline/pipeline.py --input 0 --camera-id cam_live --preview

To run tests:
  pytest tests/ -v

To check storage savings after a run:
  The pipeline prints a storage report at the end of each run.
  You can also query outputs/metadata.db directly with SQLite.

To add a new background subtraction method:
  Edit src/background_subtraction/background_subtraction.py
  Add the method name to SUPPORTED_METHODS
  Instantiate the OpenCV object in __init__

To add the enhancement module (milestone 2):
  Create src/enhancement/enhancer.py
  Implement a class that takes an image array and returns an upscaled version
  Integrate it into the pipeline after the decode step
  Real-ESRGAN CPU inference is the recommended starting point

== CONTEXT FROM EARLIER CAPSTONE RESEARCH ==

Before the sponsor meeting, the team conducted research on the broader video
compression landscape. While much of that research (drone ISR, neural codecs, Jetson
hardware) turned out to be outside the revised project scope, the following elements
from the earlier research remain relevant:

RELEVANT:
  - Background subtraction algorithms (MOG2, KNN) -- core to this project
  - Region-of-interest (ROI) compression concepts -- central to our approach
  - Open-source codec landscape (SVT-AV1, libx264, FFmpeg) -- our technology choices
  - Evaluation metrics (PSNR, SSIM, VMAF, compression ratio) -- how we measure success
  - CompressAI-Vision concept of evaluating compression impact on object detection -- 
    potentially relevant for verifying people/vehicle detection still works on compressed output
  - DoD open-source licensing policy (permissive > copyleft for DoD deployment) -- 
    shapes our library choices

NOT IN SCOPE FOR THIS PROJECT (based on sponsor meeting):
  - Drone footage or aerial/UAV content
  - Neural video codecs (DCVC, CompressAI neural models)
  - NVIDIA Jetson or GPU-accelerated inference
  - STANAG 4609 / military standards integration
  - VisDrone or UAV123 datasets (wrong content type)
  - Real-time satellite transmission


== IMPORTANT NOTES FOR COWORK ==

- All code must be Python 3.x, using only open-source libraries
- No GPU dependencies. Everything must run on CPU
- FFmpeg must be installed as a system binary (not just the Python wrapper)
- SQLite is used for metadata, no external database server needed
- Video files in data/samples/ are gitignored -- team members share test clips separately
- The working folder is capstone-compression/
- Main entry point is src/pipeline/pipeline.py
- Tests are in tests/ and run with pytest
- Notebooks go in notebooks/ for analysis and benchmarking work
